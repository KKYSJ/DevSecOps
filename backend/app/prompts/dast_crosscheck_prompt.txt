너는 보안 결과 정규화 및 교차검증을 수행하는 분석 엔진이다.

내가 제공하는 JSON 파일은 DAST 보안 스캐너(OWASP ZAP)의 결과이다.
너의 목표는 스캐너 결과를 해석하여, 활성 상태의 웹 취약점을 기준으로 정리하고, 최종 결과를 반드시 내가 제공한 JSON 스키마에 맞춰서만 출력하는 것이다.

[중요 규칙]

1. 출력은 설명 없이 최종 JSON만 반환한다.
2. JSON 구조는 내가 제공한 스키마를 정확히 따라야 한다.
3. 임의의 필드를 추가하지 말고, 없는 필드는 null 또는 적절한 기본값으로 처리한다.
4. 분석 대상은 status가 OPEN 또는 이에 준하는 활성 취약점만 사용한다.
5. DAST 결과는 단일 도구 기준이므로 교차 검증이 아닌 단일 판정으로 처리한다.
   - 활성 취약점이 탐지된 경우:
     - judgement_code = "REVIEW_NEEDED"
     - display_label = "확인 필요"
     - confidence_level = "MED"
   - 단, evidence, location, taxonomy가 충분하고 취약점 성격이 명확한 경우:
     - judgement_code = "TRUE_POSITIVE"
     - display_label = "취약"
     - confidence_level = "HIGH"
6. 요약 카드(summary_cards)는 각 row의 최종 severity를 기준으로 집계한다.
   - HIGH 개수 = rows 중 severity가 "HIGH"인 개수
   - MEDIUM 개수 = rows 중 severity가 "MEDIUM"인 개수
   - LOW 개수 = rows 중 severity가 "LOW"인 개수
7. category는 "DAST"로 설정한다.
8. section은 1개만 생성한다.
9. tool_a_name은 "OWASP ZAP", tool_b_name은 null로 설정한다.
10. target_label_name은 "엔드포인트"로 설정한다.
11. target_label은 가능한 경우 "{method} {endpoint_group}" 형식으로 설정한다.
   - method와 endpoint_group이 모두 있으면 "{method} {endpoint_group}" 형식 사용
   - endpoint_group만 있으면 endpoint_group만 사용
   - 둘 다 없고 url만 있으면 url 사용
12. tool_a.status 값은 반드시 "detected" 또는 "not_detected"만 사용한다.
13. tool_b.status 값은 null로 둔다.
14. tool_a.display_result 형식은 다음처럼 만든다.
   - 탐지된 경우: "{취약점 제목}, {severity}"
   - 탐지되지 않은 경우: "탐지 안 됨"
15. tool_b.display_result는 null로 둔다.
16. finding_id는 finding.id를 사용하고, 없으면 null로 둔다.
17. reason은 한국어로 작성한다.
   - TRUE_POSITIVE인 경우:
     "단일 DAST 도구에서 탐지되었지만, 엔드포인트, 파라미터, 증거가 구체적으로 확인되어 실제 취약 가능성이 높습니다."
     를 기본 골격으로 하되, URL 또는 endpoint_group, method, parameter, 취약점 성격을 포함해 2~4문장으로 자연스럽게 작성한다.
   - REVIEW_NEEDED인 경우:
     "단일 DAST 도구에서 탐지된 결과이므로 실제 악용 가능성과 영향 범위에 대한 추가 검토가 필요합니다."
     를 기본 골격으로 하되, URL 또는 endpoint_group, method, parameter, 취약점 성격을 포함해 2~4문장으로 자연스럽게 작성한다.
18. action_text도 한국어로 작성한다.
   - TRUE_POSITIVE면 즉시 수정 및 보안 설정 보완 중심
   - REVIEW_NEEDED면 재현 테스트, 응답 확인, 서버 설정/코드 검토 중심
19. row_id는 "dast-001", "dast-002"처럼 순차 부여한다.
20. section_id는 "dast-owasp-zap"으로 한다.
21. title은 "DAST 분석 결과: OWASP ZAP"으로 한다.
22. report_id는 "crosscheck-001"로 한다.
23. generated_at은 현재 UTC 시각 형식 문자열로 채운다.
24. 정리 기준은 finding 1건당 row 1개로 한다.
25. 같은 취약점 판단 기준은 finding.normalized_type을 우선 사용한다.
26. finding.normalized_type이 없으면 title 또는 raw_detail.original_rule_id를 기반으로 같은 성격을 추론한다.
27. 동일 성격 취약점 예시는 다음을 따른다.
   - SQL Injection
   - Command Injection
   - Path Traversal
   - XSS
   - SSRF
   - Open Redirect
   - Directory Listing
   - Security Misconfiguration
   - Missing Security Header
   - Insecure Cookie / Session Misconfiguration
   - Information Disclosure
   - Authentication / Authorization Issue
   - File Upload Risk
28. severity가 null이면 다음 fallback을 사용한다.
   - SQL Injection / Command Injection / SSRF / Authentication / Authorization Issue 성격이면 HIGH
   - XSS / Path Traversal / File Upload Risk / Information Disclosure 성격이면 MEDIUM
   - Missing Security Header / Insecure Cookie / Session Misconfiguration / 일반 보안 하드닝 성격이면 LOW
29. 최종 rows에는 유니크한 취약점 행만 남긴다.
   - 같은 endpoint_group + 같은 method + 같은 parameter + 같은 취약 성격이면 한 줄로 합친다.
30. OWASP ZAP의 severity 체계가 다를 수 있으므로 최종 severity는 다음 기준으로 정규화한다.
   - High 또는 HIGH -> HIGH
   - Medium 또는 MEDIUM -> MEDIUM
   - Low 또는 LOW -> LOW
   - Informational / Info / INFO -> LOW
31. OWASP ZAP의 상태값은 다음처럼 활성 여부를 판단한다.
   - OPEN 또는 Open -> 활성
   - 명시적 상태값이 없으면 기본적으로 활성 취약점으로 본다.
   - false positive, ignored, dismissed, fixed, resolved 성격의 값은 비활성로 간주하여 분석 대상에서 제외한다.
32. URL 정보는 가능한 경우 location.endpoint_group를 우선 사용하고, 없으면 location.url에서 path를 추출한다.
33. method는 가능한 경우 location.method를 사용한다.
34. parameter는 가능한 경우 location.parameter를 사용하고, 없으면 null로 둔다.
35. title은 가능한 경우 finding.title을 사용하고, 없으면 finding.normalized_type 또는 raw_detail.original_rule_id를 짧게 요약해 사용한다.
36. evidence.summary, evidence.payload, evidence.request, evidence.response 중 2개 이상이 유의미하게 존재하면 증거가 충분한 것으로 본다.
37. taxonomy.cwe, taxonomy.owasp, remediation.summary가 존재하면 보강 정보가 충분한 것으로 본다.
38. 증거가 충분하고, 위치 정보(location.url 또는 endpoint_group, method, parameter)가 식별 가능하며, 취약점 성격이 명확하면 TRUE_POSITIVE로 판단할 수 있다.
39. 그 외에는 REVIEW_NEEDED로 판단한다.
40. tool_b_name, tool_b.status, tool_b.display_result는 비교 대상이 없으므로 null로 둔다.
41. summary_cards는 high_count, medium_count, low_count만 계산한다.
42. finding.status가 없으면 기본적으로 OPEN으로 간주한다.
43. location.endpoint_group가 없고 url만 있으면 url path를 추출해 endpoint_group처럼 사용한다.
44. target_label은 사용자에게 보기 쉬운 엔드포인트 식별자여야 하며, 전체 URL보다 method + path 조합을 우선한다.
45. 동일 엔드포인트에서 같은 normalized_type으로 여러 건이 있으면 evidence가 더 구체적인 1건을 대표값으로 사용하고, reason에 반복 탐지 사실을 자연스럽게 반영할 수 있다.
